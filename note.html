GIT add and commit

The Staging Area is where your files go before the snapshots are taken. Not every file you modified on the Working Directory should be taken into account when taking a snapshot of the current state of the project. Only the files placed in the Staging Area will be snapshotted.

 To add a file to the Staging Area, we use the Git command “add.”
Plain Text
training@training-virtualbox:~/mynewproject$ git add readme.md 
training@training-virtualbox:~/mynewproject$ git status
On branch master

No commits yet

Changes to be committed:
  (use "git rm --cached <file>..." to unstage)
        new file:   readme.md

a commit is just a snapshot of the entire project at a certain time. it takes a picture of the entire project.
We can now commit the files we staged earlier. Before each commit, you should check the status of the Working Directory and the Staging Area. If all the files you want to commit are in the Staging Area (under the phrase “Changes to be committed”), you can commit. If not, you have to stage them with “git add.”
To commit all the changes we made, we use “git commit.” This will take a snapshot of our current state of the project.
Plain Text
training@training-virtualbox:~/mynewproject$ git commit -m "Add README.md to the project"
[master (root-commit) 156ff3a] Add README.md to the project
 1 file changed, 0 insertions(+), 0 deletions(-)
 create mode 100644 readme.md

The summary of the commit will contain a lot of information:
  • The current branch: master
  • The name of the previous commit: root-commit because this is our first commit
  • The name of the commit: the first seven letters of the commit hash
  • The commit message
  • The number of files changed: one file
  • The operation done to each file: creation
Plain Text
training@training-virtualbox:~/mynewproject$ git status
On branch master
nothing to commit, working tree clean


training@training-virtualbox:~/mynewproject$ git branch
* master

Remote Repository

a remote repository is where you can publish your project to make it available to everyone. And anyone can clone your repository, so they can follow your advancements to get the latest changes.

Initialize Remote Repositories

One of the ways to initialize remote repository is to link an existing local repository with remote repository.
The repository names can differ between local and remote, but it would be a good idea to use a unique name so you don’t get confused.
Plain Text
cd ~
mkdir demo
cd demo
git init

Now that we have our local repository, it’s time to link it to the remote! To list, add, or remove remotes, we will use the git remote command. For example, let’s link our current remotes using this command:
Plain Text
git remote

You shouldn’t get any result because it’s a brand-new repository and we haven’t linked any remote to it. 
You will need the unique link to your repository to be able to link a local repository to it; so, grab yours from the previous section. Mine is https://github.com/trainingdemook/demo.git Don’t forget the .git at the end!
You will also need to create a name for your remote repository. That way, you can have multiple remotes within a single project. It may be necessary in the case where the test and production remotes are different for each other. The default name is “origin” per convention. Although you can choose any name, it is recommended to use origin as the name of the remote where teammates share their work.
The command to add a link to a remote is simple. It’s
CSS/SCSS
git remote add [name] [link]

So, to add a link to the newly created repository, you’ll have to execute this command:
Plain Text
$ git remote add origin https://github.com/trainingdemook/demo.git

Creating a branch

use the “git branch” command followed by the branch name. Keep in mind that the branch name should only contain alphanumeric values and dashes or underscores; no spaces allowed.
Plain Text
$ git branch <name>

For example, let’s create a development branch for our project. Let’s name it “develop.” Here’s how to do it:
Plain Text
training@training-virtualbox:~/branch-demo$ git branch
* main
training@training-virtualbox:~/branch-demo$ git branch develop
training@training-virtualbox:~/branch-demo$ git branch
  develop
* main

After you execute that command, you will notice that nothing has changed in your project. That’s because creating a branch is just about creating a reference to the last commit of the current branch and nothing else. To begin working with a branch, you have to switch to it
The current branch is pointed with "*" symbol

Deleting a branch

To delete a branch, simply use the same command as to create one but with the option “-d.”
$ git branch -d <name>\
Let us delete "develop" branch.Please make sure to switch to different branch from the branch to be deleted.
SQL
training@training-virtualbox:~/branch-demo$ git branch
* develop
  main
training@training-virtualbox:~/branch-demo$ git branch -d develop
error: Cannot delete branch 'develop' checked out at '/home/training/branch-demo'
training@training-virtualbox:~/branch-demo$ git checkout main
Switched to branch 'main'
Your branch is up to date with 'origin/main'.
training@training-virtualbox:~/branch-demo$ git branch -d develop
Deleted branch develop (was d5cb481).
training@training-virtualbox:~/branch-demo$ git branch
* main
training@training-virtualbox:~/branch-demo$
